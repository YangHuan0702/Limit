# 零拷贝 I/O：从应用内存到网络的全路径分析

## 什么是"真正的零拷贝"

大多数声称"零拷贝"的系统实际上只是做到了"减少拷贝"。在 3FS 的数据读取路径上，真正的零拷贝意味着：

```
磁盘 → （DMA）→ 内核页缓存 → （拷贝）→ 用户缓冲 → （拷贝）→ 网卡
```
变成：
```
磁盘 → （DMA）→ RDMA 注册内存 → （RDMA READ）→ 客户端内存
```

消除了所有 CPU 参与的内存拷贝操作。

## 3FS 的零拷贝设计思路

### 问题的本质

传统存储服务的数据路径：

1. 客户端 `recv()` → 内核缓冲 → 用户缓冲（第 1 次拷贝）
2. 用户缓冲 → 应用处理
3. 磁盘读取 → 内核页缓存（DMA，无 CPU 拷贝）
4. 内核页缓存 → 用户缓冲（第 2 次拷贝）
5. 用户缓冲 → 网络发送缓冲（第 3 次拷贝）
6. 发送缓冲 → 网卡（DMA）

3FS 通过以下机制逐步消除这些拷贝：

### 消除网络拷贝：RDMA

服务端分配好 RDMA buffer，把它的地址（`RDMARemoteBuf`）告诉客户端。客户端发起 RDMA READ，网卡直接把服务端内存的数据推到客户端的注册内存，全程没有 CPU 参与的拷贝。

### 消除磁盘到内存的拷贝：绕过页缓存

3FS 存储服务使用 O_DIRECT 打开 chunk 文件：

```cpp
// 绕过内核页缓存，数据直接 DMA 到用户指定的 RDMA buffer
int fd = open(chunkPath, O_RDONLY | O_DIRECT);
```

配合 `libaio` 或 `io_uring`，磁盘 I/O 可以直接把数据写入已注册的 RDMA buffer，省掉内核页缓存的中间层。

**O_DIRECT 的对齐要求**：
- 用户缓冲地址必须按照 sector size（通常 512B 或 4096B）对齐
- 读写偏移和长度也必须按 sector size 对齐
- 这是 `AioReadJob` 中 `headLength` 和 `tailLength` 字段存在的原因

```cpp
struct AioReadJob {
    size_t headLength;   // 头部对齐 padding（不属于实际数据）
    size_t tailLength;   // 尾部对齐 padding
    size_t readLength;   // 实际读取长度（包含 padding）
    size_t readOffset;   // 对齐后的起始偏移
};
```

## 内存注册：性能的根基

### ibv_reg_mr 的代价

每次调用 `ibv_reg_mr` 需要：
1. 把物理页 pin 住（防止被换出）
2. 建立 IOMMU 映射（让 RNIC 能访问这段物理内存）
3. 注册到内核的 MR 表（有全局锁竞争）

在高频 I/O 路径上，这个开销是不可接受的。

### 3FS 的解决方案：预分配池

```
启动时：
┌─────────────────────────────────────┐
│  ibv_reg_mr(pd, bigBuffer, size, flags) │
│  一次注册整块大内存                      │
└─────────────────────────────────────┘

运行时：
┌──────────┐  ←  RDMABufPool::allocate()
│  buf[0]  │
├──────────┤
│  buf[1]  │  每个 buf 是大内存的一个切片
├──────────┤  共享同一个 MR（rkey 相同）
│  buf[2]  │
└──────────┘
```

关键点：切片 buffer 与父 MR 共享同一个 `rkey`，只需要一次注册。`RDMABuf` 通过 `begin_` 和 `length_` 指向 `Inner` 内存的子范围，而不复制数据。

### 多设备 MR 共享

```cpp
class RDMABuf::Inner {
    std::array<RDMABufMR, IBDevice::kMaxDeviceCnt> mrs_;
    // mrs_[0] = 设备 0 上的注册
    // mrs_[1] = 设备 1 上的注册
    // ...
};
```

同一块物理内存注册到所有可用 RDMA 设备。选择哪个设备发送时，只需要索引对应的 `mrs_[devId]` 取到 lkey/rkey，而不需要重新注册。

## USRBIO：用户态旁路 I/O

### FUSE 的瓶颈

即使数据路径完全零拷贝，FUSE 本身仍有开销：

1. 应用 `read()` 系统调用进入内核
2. 内核 FUSE 模块把请求放入 `/dev/fuse` 队列
3. FUSE daemon（用户态）从队列取出请求
4. 执行实际 I/O
5. 把结果写回 `/dev/fuse` 队列
6. 内核把结果拷贝给应用

每个 `read()` 都有两次内核/用户态切换和一次数据拷贝（步骤 6）。

### USRBIO 的设计

USRBIO（User-space Bypass I/O）用共享内存绕过这个路径：

```
┌──────────────────────────────────────────────────┐
│                  应用进程                         │
│                                                  │
│  IoBuffer (共享内存 Iov)  ←─── 数据在这里       │
│  IoRing   (共享内存 Ior)  ←─── 提交/完成队列    │
│                                                  │
│  sqe = {fd, offset, len, buf}                    │
│  ring.submit(sqe)  ──────────────────────────┐  │
└──────────────────────────────────────────────│──┘
                                               │
┌──────────────────────────────────────────────▼──┐
│              3FS Native Client                   │
│                                                  │
│  ioRingWorker() 轮询 ring                        │
│  批量取出 sqe → 组装 StorageClient 请求         │
│  RDMA 读取数据直接写入 IoBuffer (共享内存)       │
│  写入完成 cqe 到 ring                           │
└──────────────────────────────────────────────────┘
```

**关键优化**：
- 应用和 native client 共享同一块物理内存（`IoBuffer`）
- 数据直接通过 RDMA 写入共享内存，应用直接读取，**没有任何拷贝**
- 多个 sqe 自动批量化，一次 RPC 可以携带多个 I/O 请求

### IoRing 数据结构

```
┌────────────────────────────────────────┐
│  ringSection (元信息 + 同步变量)       │
├────────────────────────────────────────┤
│  SqeSection  (提交队列)                │
│  ┌─────┬─────┬─────┬─────┐            │
│  │sqe0 │sqe1 │sqe2 │ ... │            │
│  └─────┴─────┴─────┴─────┘            │
│           ↑ producer 写                │
├────────────────────────────────────────┤
│  CqeSection  (完成队列)                │
│  ┌─────┬─────┬─────┬─────┐            │
│  │cqe0 │cqe1 │cqe2 │ ... │            │
│  └─────┴─────┴─────┴─────┘            │
│           ↑ native client 写           │
└────────────────────────────────────────┘
```

### 批处理策略

```
io_depth = 32    // 积累 32 个请求后批量提交
timeout = 1ms    // 未满 32 个但超过 1ms 也提交
```

这两个参数的权衡：
- `io_depth` 越大，每次 RPC 携带的请求越多，网络效率越高，但延迟越高
- `timeout` 越小，延迟越低，但小 batch 的 RPC 开销比例越大
- 对于大文件顺序读（训练场景），较大的 `io_depth` 更合适

## 用户 Buffer 注册（进阶零拷贝）

```cpp
// 应用把自己的内存注册为 RDMA buffer
auto rdmaBuf = RDMABuf::createFromUserBuffer(myBuffer, size);

// 告诉 storage 节点：把数据直接写到 myBuffer 这个地址
storageClient.rdmaRead(remoteBuf, rdmaBuf);
```

这是最彻底的零拷贝：应用的目标内存直接参与 RDMA 操作，数据到达时就已经在应用需要的地方了。

**前提条件**：
1. 应用内存必须对齐（至少 4KB，推荐 2MB 大页对齐）
2. 内存不能被 swap 换出（需要 `mlock` 或 `MAP_LOCKED`）
3. 内存大小必须覆盖整个 I/O 范围

## 对齐与大页

### 为什么对齐很重要

RDMA 操作和 O_DIRECT I/O 都需要内存对齐。不对齐的后果：
- O_DIRECT：`EINVAL` 错误
- RDMA：RNIC 需要额外处理跨页边界的操作，性能下降

### 推荐的内存布局

```
2MB 大页对齐的 RDMA buffer pool：
┌────────────────────────────────────────────────┐
│           2MB Huge Page                         │
│  ┌──────┬──────┬──────┬──────┬──────┬──────┐  │
│  │64MB  │64MB  │64MB  │ ...  │      │      │  │
│  │chunk │chunk │chunk │      │      │      │  │
│  └──────┴──────┴──────┴──────┴──────┴──────┘  │
└────────────────────────────────────────────────┘
```

使用 `mmap(MAP_HUGETLB)` 分配大页，减少 TLB miss，同时满足 O_DIRECT 对齐要求。

### ibv_reg_mr 的大页优化

```
普通页：每个 4KB 页一个 IOMMU 条目
大页：每个 2MB 大页一个 IOMMU 条目
```

大页大幅减少 IOMMU 表的大小，在内存密集型 RDMA 工作负载下，IOMMU TLB 命中率显著提升。

## 实现优先级建议

如果你正在自己实现，建议按以下顺序推进：

1. **先做内存注册池**：把 `ibv_reg_mr` 从关键路径移出去
2. **再做批量 WR 提交**：把单次 RDMA 操作替换为 batch
3. **再做 O_DIRECT + 对齐**：消除内核页缓存拷贝
4. **最后做 USRBIO**：实现真正的零拷贝端到端路径

每一步都可以独立验证性能收益，避免过早复杂化。
