**问题**：一个请求一个请求处理，GPU大部分时间在空转。

复制

```
请求1：[处理中...] [等待] [等待] [等待]
请求2：[等待] [处理中...] [等待] [等待]
请求3：[等待] [等待] [处理中...] [等待]
```

**解法**：把多个请求打包成一批，一起喂给GPU。

复制

```
请求1,2,3：[一起处理中...] [一起处理中...]
```

**两种批处理策略**：

|策略|原理|适用场景|
|---|---|---|
|**Static Batching**|等凑够N个请求再一起处理|离线批量推理|
|**Continuous Batching**|请求来了就进批次，生成完就走，不等人|在线服务|

**Continuous Batching是vLLM的核心创新之一。**

#### 3. vLLM —— 生产级推理框架

**vLLM解决了什么？**

|问题|vLLM的解法|
|---|---|
|KV Cache显存碎片化|**PagedAttention**：把KV Cache分成固定大小的block，按需分配，像内存分页|
|批处理效率低|**Continuous Batching**：动态调度，请求随进随出|
|并发受限|**高并发支持**：单卡支持数千并发请求|

**PagedAttention图解**：

复制

```
传统KV Cache：每个请求预分配一大块连续显存
请求1：[████████████░░░░░░░░]  用了一半，浪费一半
请求2：[████░░░░░░░░░░░░░░░░]  用了1/4，浪费3/4

PagedAttention：按需分配block，像内存分页
请求1：[block1][block2][block3]  用多少占多少
请求2：[block4][block5]          用多少占多少
```
#### 4. 其他推理优化技术

| 技术                       | 原理                        | 效果              |
| ------------------------ | ------------------------- | --------------- |
| **Flash Attention**      | 优化注意力计算的内存访问模式            | 速度提升2-4x，显存占用降低 |
| **Speculative Decoding** | 小模型猜几个token，大模型验证         | 延迟降低2-3x        |
| **Prefix Caching**       | 缓存公共前缀（如system prompt）的KV | 相同前缀的请求直接复用     |

### 核心概念
#### LoRA —— 低成本微调

**问题**：全量微调70B模型，需要更新700亿个参数，显存爆炸。

**LoRA的思路**：冻结原始模型不动，只训练一个很小的"旁路矩阵"。
```
原始权重 W（冻结不动）：
输入 x → W → 输出

LoRA：
输入 x → W（冻结）→ 输出
输入 x → A → B（可训练）→ 输出
                              ↑
                    最终输出 = W(x) + B·A(x)

```
**关键参数**：

- **rank ®**：旁路矩阵的维度，越大表达能力越强但参数越多。常用 r=8~64
- **alpha**：缩放系数，控制LoRA贡献的强度

**效果**：

- 只训练 0.1%~1% 的参数
- 显存需求降低 3~10 倍
- 效果接近全量微调

**适用场景**：让通用模型学会特定任务/领域/风格

#### 量化 —— 压缩模型

**原理**：把模型权重从高精度降到低精度。

复制

```
FP32（32位浮点）：每个参数占 4 字节
FP16（16位浮点）：每个参数占 2 字节，显存减半
INT8（8位整数）：每个参数占 1 字节，显存再减半
INT4（4位整数）：每个参数占 0.5 字节，70B模型只需35GB
```

**量化方式对比**：

| 方式             | 原理                 | 精度损失 | 速度    |
| -------------- | ------------------ | ---- | ----- |
| **PTQ（训练后量化）** | 直接把训练好的模型权重量化      | 稍大   | 快     |
| **GPTQ**       | 基于Hessian矩阵的智能量化   | 小    | 快     |
| **AWQ**        | 保护重要权重通道           | 更小   | 快     |
| **GGUF**       | llama.cpp生态，CPU也能跑 | 小    | CPU友好 |
**实际效果**：

- 7B模型 INT4量化后只需 ~4GB 显存，消费级显卡就能跑
- 精度损失通常在 1-2% 以内，大部分场景可接受

#### 3. 蒸馏 —— 大模型教小模型

**原理**：用大模型（Teacher）的输出作为训练目标，训练小模型（Student）。
```
Teacher（70B）→ 生成大量高质量回答
                        ↓
Student（7B）→ 学习模仿Teacher的回答

```

**什么有效？**

- 大模型的"软标签"比原始标签信息更丰富
- 小模型学到了大模型的推理方式，而不只是最终答案

**适用场景**：

- 需要部署到边缘设备（手机、IoT）
- 需要极低延迟
- 需要大幅降低推理成本

```
需要模型做特定任务？
  ├── 是 → LoRA微调（让模型学会你的任务）
  └── 否 → 模型够用吗？
              ├── 太大跑不动 → 量化（压缩模型）
              └── 太贵 → 蒸馏（用小模型替代）
```

#### LLM评测—— 怎么衡量模型好坏？

**评测维度**：

|维度|衡量什么|例子|
|---|---|---|
|**质量**|回答准不准、有没有幻觉|“法国首都是哪？” → “巴黎” ✅|
|**安全性**|有没有有害内容|拒绝教人做炸弹|
|**性能**|速度快不快、稳不稳定|TTFT < 500ms, QPS > 50|
|**成本**|花了多少钱|单次对话成本 < ¥0.01|

**评测方法**：

|方法|原理|优缺点|
|---|---|---|
|**人工评测**|人打分|最准但最慢、最贵|
|**自动评测（LLM-as-Judge）**|用GPT-4当裁判给回答打分|快、便宜，但裁判本身可能不靠谱|
|**Benchmark**|标准化测试集（MMLU、GSM8K、HumanEval）|可对比、可复现，但和实际业务可能有差距|
|**A/B测试**|两个版本同时上线，看用户反馈|最真实，但成本高、周期长|

#### 架构设计—— 生产级LLM系统长什么样？
```
用户请求
   ↓
┌─────────────┐
│  API Gateway │ ← 限流、鉴权、路由
└──────┬──────┘
       ↓
┌─────────────┐
│  Orchestrator│ ← 判断走哪条链路
└──┬──┬──┬───┘
   ↓  ↓  ↓
 RAG Agent 直接回答 拒绝
   ↓  ↓
┌─────────────┐
│  LLM Service │ ← vLLM / API调用
└──────┬──────┘
       ↓
┌─────────────┐
│  后处理      │ ← 安全过滤、格式化、日志
└──────┬──────┘
       ↓
   返回用户

```

原则	说明
降级策略	主模型挂了自动切备用模型，API超时返回缓存答案
缓存	相同问题直接返回缓存，省token省时间
异步处理	耗时任务（文档索引、批量推理）丢消息队列
可观测性	每个请求全链路trace，出问题能快速定位
