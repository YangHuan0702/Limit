
#### 推理优化: 让LLM跑得更快、更省、扛得住更多用户。

在学vLLM的时候我们有了解到其中有个KV Cache的东西，它加快了QPS；常规的流程是一个请求过来就直接预先占用了一部分的显存，请求一多线程就不够用了，而vLLM则是通过将显存以块的形式去申请使用，具体流程是vLLM启动后会申请一部分的显存做为KV内存池（通过`--gpu-memory-utilization`参数控制），之后不管来多少用户请求，他都在这个池子中按块去精细化分配，不会出现过多占用和浪费的情况，以达到最大化并发。推理优化其实是对模型的启动参数进行一些设置来保证模型的回答能够更加符合预期； 

那这跟vLLM有啥关系？ 就像上面写的KV Cache一样，为什么要说KV Cache就得说说KV给我们带来了的最大好处是啥？ 答案很明显是并发大了，这就好比是银行的柜台，以前只能分成3个队伍，但是现在可以分成5个队伍了；  我想说的是推理优化并不单单是模型本身的回答准确度，还有使用层面的一些问题。

既如此，推理优化我们需要学的就是vLLM的一些参数了；

**显存与性能调优核心：**
- `--gpu-memory-utilization`：控制 vLLM 启动时预先霸占的显存比例。默认是 0.9，如果报错 OOM，就下调到 0.85 或 0.8。
- `--enable-prefix-caching`：**强烈建议开启。** 开启前缀缓存，对于带有长 System Prompt 的多轮对话，能极大省下重复计算的时间和显存。
- `--max-num-seqs`：这辆“公交车” 🚌 的最大载客量（最大并发请求数）。如果你的单条请求很长，可以适当调小这个值以防运行中途 OOM。默认 256。
- `--enforce-eager`：**生产环境请去掉！** 强制关闭 CUDA Graphs 加速。只有在排查一些诡异的底层 BUG 时才打开它。

需要着重说明一下的是`--enable-prefix-caching`开启前缀缓存这个参数，LLM 每次生成回复，都要对输入的所有 token 做 Attention 计算，计算结果就存在 KV Cache 里。

**模型与上下文控制：**
- `--max-model-len`：限制模型的最大上下文长度。如果你的业务最多只用到 4000 字，显式设置为 4096 可以避免 vLLM 按照模型原生的 32K 去预留天量的 KV Cache 显存。 
- `--dtype`：加载权重时使用的数据类型。通常设为 `half`、`float16` 或 `bfloat16`。模型加载进显存时，每个数字用多少位来表示；模型本质上就是几十亿个浮点数（参数）。这些数字存在显存里，`--dtype` 决定**每个数字占多少 bit**。
	- float32 → 32 bit 存一个数 → 最精确，但显存占用最大
	- float16 → 16 bit 存一个数 → 精度略降，显存砍半
	- bfloat16 → 16 bit 存一个数 → 和 float16 一样大，但数值范围更广，训练更稳
	- **对推理来说，float32 和 float16 的输出质量几乎没有肉眼可见的差别**，所以直接用 float16 或 bfloat16 就行，没必要用 float32 浪费显存。（推理对精度的要求不需要太高，训练则需要高精度。）
- `--quantization`：指定量化格式（比如 `awq`, `gptq`, `fp8`）。把数字从 16 bit 再压到 4 bit，`--dtype` 最低到 16 bit，而量化可以压到 **4 bit 甚至更低**。听起来误差很大，但量化技术的聪明之处在于——**它会找到一种映射方式，让压缩后的数字在实际计算时误差尽量小**，而不是简单粗暴地截断。所以实际用下来，4 bit 量化的模型回答质量和原版差距很小，但显存能省掉 75%。

`--quantization`与`--dtype`的区别：两个参数的本质都是同一件事：**用少量精度损失换取更小的显存占用**，区别只是压缩力度不同。

|      | --dtype       | --quantization |
| ---- | ------------- | -------------- |
| 压缩程度 | 温和（32→16 bit） | 激进（16→4 bit）   |
| 显存节省 | 50%           | 75%            |
| 质量损失 | 几乎没有          | 轻微但可接受         |
| 使用方式 | 启动时指定即可       | 需要提前下载量化版模型    |

另外就是到处都在谈的上下文到底是什么鸡儿？ 从字面意思来理解的是不是对的？
**上下文：整个会话的 token 总长度。具体来说是这一次请求里塞进去的所有文字**，包括：
```
System Prompt + 历史对话（所有轮次）+ 当前用户的问题 + 模型即将生成的回答
```

#### 模型优化
模型优化的核心目的在于：如何在不加显卡的情况下，让模型体积更小，或者让它更懂你的特定业务。
在这个阶段，我们需要掌握两项高优技术：
- **量化 (Quantization)：** 就像把几十兆的无损原图压缩成几兆的 JPEG。我们将模型原本的高精度参数（比如 16 位浮点数）压缩成 8 位甚至 4 位。代价是极微小的精度损失，但收益是模型显存占用直接减半，让你能在有限的资源里塞进更大的并发。主流格式有 AWQ、GPTQ 和 FP8。
- **LoRA (低秩微调)：** 传统的微调需要更新模型成百上千亿的参数，成本极高。LoRA 则是不动原来的“主脑”，只在旁边外挂一个非常小的“外接学习模块”（适配器）。结合 Unsloth 加速框架，你在单卡上几个小时就能练出专属模型，而且这个外接模块可以随时热插拔。


##### 量化
核心理论：量化 (Quantization) 到底是啥？简单来说，量化就是**降低模型权重的“分辨率”**。 大模型默认的参数是 16位浮点数（FP16/BF16）。量化就是把这些数字强行压缩成 8位（INT8/FP8）甚至 4位（INT4）。
- **收益**：显存占用直接减半甚至砍掉 70%！这意味着同样一张卡，原来只能跑 100 个并发，现在能跑 300 个。
- **代价**：模型会变“笨”一点点（精度损失），但在主流的 AWQ 或 GPTQ 算法下，这种损失在日常业务中肉眼几乎看不出来。
**直接拉取官方已经量化好的模型！** 绝大多数开源模型（包括 Qwen）都会官方发布 AWQ 或 GPTQ 版本。
```
# 注意看 model 名称变了，且多加了 --quantization awq
VLLM_USE_MODELSCOPE=TRUE 
python -m vllm.entrypoints.openai.api_server  --model "Qwen/Qwen2.5-VL-3B-Instruct-AWQ" --quantization awq --port 8100 --max-model-len 4096 --gpu-memory-utilization 0.8 --enable-prefix-caching
```
注：vLLM 底层会自动识别 `--quantization awq` 并调用专门的优化算子（如 Marlin 算子）来极速加载和推理这些 4 位参数。

`--quantization`参数：
- GPTQ：暴力但经典的“老大哥” (4-bit)
	- **原理**：这是最早流行起来的 4位量化技术。它就像是一个无情的压缩机，通过复杂的数学计算，硬生生把 16位的数据压成 4位。
	- **优缺点**：压缩率极高，模型体积直接缩小 70%。但因为它比较暴力，有时候会把模型里一些非常关键的“神经元”也给压坏了，导致模型在某些复杂逻辑上变笨（专业术语叫“精度掉点”）。
- AWQ：聪明的“重点保护派” (4-bit) —— **当前最推荐的 4位方案**
	- **原理**：AWQ（Activation-aware Weight Quantization）是在 GPTQ 之后出的改良版。它在压缩前会先观察模型，发现模型里其实只有 1% 的参数是“VIP”（对最终结果影响极大）。于是它把这 1% 的 VIP 保护起来保持高精度，只去疯狂压缩剩下 99% 的普通参数。
	- **优缺点**：它既做到了和 GPTQ 一样的极限压缩比，又极大程度保留了模型原本的聪明才智。**现在你在 HuggingFace 或 ModelScope 上看到的开源模型，首选下载 AWQ 版本。**
- FP8：有钱人的“原生黑科技” (8-bit)
	- **原理**：这是英伟达在最新的架构（比如 Hopper 架构的 H100、Ada 架构的 RTX 4090）上强推的硬件级 8位浮点数格式。它不搞复杂的软件压缩，而是 GPU 硬件原生就支持这种数据格式。
	- **优缺点**：速度极其狂暴，精度几乎和原始 16位一模一样，0 脑损！但致命缺点是：**挑硬件**。如果你租的服务器是老显卡（比如 V100、A10、或者 RTX 3090），强行跑 FP8 要么报错，要么比原来还慢。
- NF4 (BitsAndBytes) —— 炼丹微调（LoRA）的绝对神兵
	- **本质是什么**：NormalFloat 4-bit。这玩意儿**不是为了“部署接客”设计的，而是为了“训练微调”设计的**。
	- **为什么你要懂**：LoRA 微调这就是核心！ **QLoRA（量化低秩微调）**，底层用的就是它。它能在你微调训练时把模型死死压在极小的显存里，让你一张普通的显卡就能训练几十亿参数的模型。
- GGUF (原 GGML) —— 没显卡玩家的狂欢，服务器的毒药
	- **本质是什么**：专门为 **CPU**（尤其是苹果 Mac 的 M 系列芯片）量身定制的格式。
	- **避坑指南**：你在 Hugging Face 上搜索模型，会看到海量带有 `-GGUF` 后缀的文件。**千万别把它下载到你的 vLLM 服务器上！** vLLM 是重度依赖英伟达 GPU 的，强行加载 GGUF 格式会让系统懵逼，效率极低。这是留给本地没有好显卡的开发者玩的。

| -------------- | ----------- | --------------- | ------------------------ |

常见坑点与规避方法:
- **坑点 1：乱用量化格式。** 很多人随便下个 GGUF 格式就往 vLLM 里塞，vLLM 对 GGUF 的支持主要在 CPU/边缘侧。**规避：** 服务器端（GPU）永远首选 AWQ、GPTQ 或最新的 FP8。
- **坑点 2：忽略硬件支持。** 如果你租的是很老的卡（比如 V100），跑 AWQ 可能会比不量化还慢，因为老卡没有专门处理 INT4 的硬件指令。**规避：** 确保你的卡是 RTX 30/40 系或 A10/A100/H100。

##### 微调

在这个环节，我们要用最低的成本，把一个通用的 Qwen 模型，变成完全贴合你业务形态的“专属专家”。

核心理论：LoRA 与 QLoRA 到底是怎么运作的？传统的“全量微调（Full Fine-Tuning）”就像是你要修改一本很厚的百科全书，你得把整本书重新排版重印一次，这需要极其庞大的显存和算力。
- **LoRA (Low-Rank Adaptation)** 的思路极其天才：它**锁死**了原模型（那本百科全书）所有的参数，不允许改动。而是在模型旁边**外挂**了几个非常小的“便利贴”（低秩矩阵，Rank $r$）。模型在思考时，会结合书本内容和便利贴上的内容一起输出。训练时，我们只更新这些便利贴！
- **QLoRA**：则是结合了我们上一节学的量化技术（使用 NF4 格式）。把底座那本百科全书压缩成 4-bit 塞进显存，然后再贴上 16-bit 的 LoRA 便利贴进行训练。这就是你能在一张普通的消费级显卡上微调大模型的终极秘密。
