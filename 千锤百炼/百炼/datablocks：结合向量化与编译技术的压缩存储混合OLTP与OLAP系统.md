本研究旨在减少高性能混合联机事务处理与联机分析处理数据库的主内存占用，同时保持高查询性能和事务吞吐量。为此，论文提出了一种面向冷数据的创新压缩列存储格式，称为数据块。
论文中指出，目前高性能分析系统要么采用向量化查询执行，要么采用即时（JIT）查询编译技术。

在OLAP中，这些系统都是采用压缩列存储格式存储数据，其查询评估的CPU效率相较于传统行存储数据库系统提升了超过一个数量级。查询评估效率的大幅度提升通常通过“向量化”执行实现。
![[Pasted image 20260408150918.png]]
向量化：通过SIMD指令达到的高效。
JIT： 避免查询解释以及其相关的开销。

论文目标：通过引入压缩技术，在保留原系统OLTP和OLAP性能的前提下，降低Hyper系统的主内存占用，为了实现这个目标，本文做出了以下贡献：
- 数据块：一种面向混合数据库系统的新型压缩列存储格式
- 针对压缩数据的轻量级索引，以提升扫描性能
- 用于谓词求值的单指令多数据（SIMD）优化算法
- 通过向量化处理，在即时编译的逐元组查询引擎中集成多种存储布局组合的设计方案
![[Pasted image 20260408151737.png]]

设计混合数据库系统非常有挑战，因为这两种类型工作负载的技术要求情况各不相同，且许多基础物理优化之间存在矛盾：
- 压缩技术可以减少内存消耗，还能因带宽使用量降低而提升OLAP性能，但是OLTP要避免使用压缩，以确保对单个元组的快速访问

为了能按元组位置高效访问单个记录，我们在数据块中仅采用轻量级压缩方案。尽管许多联机分析处理（OLAP）系统采用位打包技术来实现更高的压缩比，但我们并未采用这一方式，在论文的后面实验表明，只有当早期筛选的选择性极强，以至于没有符合条件的元组且可以跳过整个数据块或者数据块中的所有元组都符合条件时，扫描操作才能实现极快的运行速度。若早期筛选得到的符合条件的元组位置比较稀疏，对扫描列进行位解包的开销将远远超过这些算法带来的性能提升。

为了加快数据块的扫描速度，我们引入了一种新型的“位置型”小型物化聚合数据（PSMAs）。PSMAs是轻量级索引，即便无法基于物化的最大值和最小值跳过整个数据块，它们也能缩小块内的扫描范围。


 逐元组即时编译的优势在于，其生成的代码在处理每个元组所需的CPU指令数极少；能同时为OLAP与OLTP查询提供极高的效率。向量操作通过内存在不同运算间传递数据，而逐元组即时编译则通过CPU寄存器传递数据，从而节省了对性能至关重要的加载/存储指令。向量化对OLTP完全没有带来CPU效率提升，因为其效率依赖于同时对多个元组执行表达式运算，而OLTP查询仅涉及极少元组，且通常会避免全表扫描。

基于数据块选择不同的压缩方案，对即时编译的逐元组查询引擎构成了挑战。由于每个压缩块可能拥有不同的内存布局，**为一次扫描所需编译的代码路径数量会呈指数级增长**，这会导致即时编译耗时过长，无法满足即席查询和事务处理的需求。


#### 数据块
数据块是一种**自包含**的容器，以字节可寻址的压缩格式存储一个或多个属性块。数据块的设计目标是节省内存, 同时还保持了出色的OLTP和OLAP性能。
在数据库存储领域，**“自包含”（Self-contained）**是一个极其核心的设计哲学。简单来说，如果一个数据块是自包含的，就意味着：**只要你把这块内存/磁盘数据拿给执行引擎，它不需要去查阅任何外部信息（比如其他数据页或全局字典），就能独立完成解压、解析和过滤。**
![[Pasted image 20260408163846.png]]
数据块的特性：
- 不可变（一旦记录被打包到数据块中，其中的数据便不允许修改； 仅仅允许删除，此时会通过标记来标识被冻结的记录。更新操作会在内部被转换为先删除再向关系中的hot未压缩数据块中插入新纪录的形式）
- 根据数据块内属性的实际值分布选择最优的压缩方法，这能实现极高的压缩比
- 仅采用可按字节寻址的压缩格式，摒弃BitWeaving(比特交织)这类亚字节编码方式，以支持高效的随机访问
- 利用SIMD技术对压缩后的数据表示形式评估可搜索的扫描约束（等于、属于、大于、小于、范围等），以匹配符合条件的数据。由于数据块中大部分压缩数据以1字节、2字节、4字节等整数形式存储，我们的SIMD算法相比未压缩数据能实现更高的加速效果。
- 小型物化聚合数据(Small Materialized Aggregates SMAs)，其中记录了数据块内每个属性的最大/最小值，可用于判断扫描过程中是否可以跳过该数据块。
- 全新的轻量级索引结构PSMAs，它将一个值映射到该值在数据块的一系列位置。借助PSMAs即使在值域较大且无法跳过整个数据块的情况下，扫描范围也可以进一步缩小。
SMAs与PSMAs索引不局限于我们的数据块格式，也可以用于未压缩的数据块。不过在HyPer中我们避免了将这些用于hot未压缩数据，因为维护最小值与最大值信息以及更新PSMA索引量对事务处理性能产生极大的负面影响。数据块的热部分通常规模较小，因此额外的开销并不合理。


##### 数据块存储布局

1 - 数据块的第一个值是其包含的记录数，通常我们在一个数据块中存储最多2^16 条记录。

2 - 每个属性的统计摘要（SMA）。基础统计要包括存储在数据块中的每个属性的最大值与最小值。

![[Pasted image 20260408201623.png]]
为了进一步缩小扫描范围，我们在属性的基本最大/最小值信息基础上，扩展了一种轻量级索引结构，称为SMA（PSMA）。在内部，PSMA包含一个简洁的查找表，该表在冷数据块“冻结“，为数据块时生成。每个表项都会有一个扫描范围`[b,e)`。

在块中我们存储了压缩后的列数据。如果查询`where col = 21`，最笨的办法就是解压整个块并扫描全部行。PSMA的目标是预先计算一个很小的查找表，告诉查询引擎：“值等于21的行，只可能出现在第100到第150个元素之间”。这样只需要解压扫描这一小段，从而加速查询。


PSMA 的思路——建一张"值→位置"的地图
PSMA 想做的事情很直觉：

> **"我提前记录下，每个值大概在数据的哪个位置区间里"**

这样查询时，不用全扫，直接去那个区间找就行。
但问题是：值的范围可能很大（比如 4 字节整数有 2³² 种值），不可能每个值存一条记录，**表会太大**。
所以 PSMA 做了一个折中：**用 1024 个格子，把所有值映射进去**，每个格子记录"属于这个格子的值，出现在数据的哪个位置范围"。
直接用值来分配格子有个问题——**值分布不均匀**。
比如一列数据全在 1000~1010 之间，只有 11 种不同的值，但如果按原值分格子，这 11 个值会全挤在 1024 个格子里的同一个角落，**大量格子浪费，精度很差**。
用 **Δ = v - min** 之后，这 11 个值的 delta 是 0~10，**刚好均匀分布在表的前面**，每个值独占一个格子，精度最高。
这就是用 delta 的核心原因：**把值"平移"到从 0 开始，充分利用表的空间。**

索引公式 i = Δ̄ + r×256 是什么意思？
这是把 delta 值"压缩"进 1024 个格子的具体方法。
把 delta 看成字节序列，比如：
- delta = 5 → 只有 1 个字节：`[05]` → 最高非零字节 Δ̄=5，剩余字节数 r=0 → **i = 5**（独占格子）
- delta = 300 = 0x012C → 2 个字节：`[01][2C]` → 最高非零字节 Δ̄=1，r=1 → **i = 1 + 256 = 257**（和其他 256 个 delta 共享这个格子）
- delta = 70000 = 0x011170 → 3 个字节：`[01][11][70]` → Δ̄=1，r=2 → **i = 1 + 512 = 513**（和更多 delta 共享）
规律就是：**delta 越小，独享格子的可能越大，精度越高；delta越大，共享同一格子的值越多，精度越低。**
为什么说有这么个规律？ **本质上来说就是`Δ̄ `等于最高非零字节，它的变化等于其他字节序列的乘积。**
