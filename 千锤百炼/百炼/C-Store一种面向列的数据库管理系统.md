列存架构的核心优势在于：查询仅需读取所需的列，避免加载无关的属性。在涉及大量数据聚合的仓库场景中，列存性能优势显著。
传统关系型数据库会将属性按照字节/字边界对齐，并以原生格式存储。过去认为内存中对数据进行移位对齐的开销过高，但如今CPU速度增速远快于磁盘带宽，因此用充裕的CPU周期换稀缺的磁盘带宽成为合理的选择，这一权衡在以读为主的环境中尤为划算。
列存通过两种方式用CPU节省磁盘带宽：
- 数据编码：将元素压缩为更紧凑的形式（如用6位表示美国州名，替代16位缩写或更长字符串）
- 密集打包：直接将N个k位值紧凑存储为N x k位，无需对齐填充。
对于计算与存储密集型数据库应用，**低成本的刀片 / 网格计算机是最优硬件选择（廉价的商品服务器）**。因此，新型数据库架构应面向网格环境设计：G 个节点，每个节点拥有独立磁盘与内存。我们采用 ** 无共享（Shared Nothing）** 架构，将数据水平分片到各节点磁盘。未来网格计算机可扩展至数十至数百节点，系统需为此规模设计。节点可物理集中或分集群部署，因此自动分配数据结构至关重要。同时，数据水平分片便于查询内并行执行。

在C-Store论文中提到了两种存储器，分别是RS/WS：
- RS(Read-optimized Store)：它是静态的。数据是按列存储、完全排序且高度压缩的，它不支持原地更新，只能通过周期性的批量合并来产生新的版本
- WS(Writeable Store): 它是`动态的`。虽然它在逻辑上也是列式，但是在物理上是一个B+树。每个列其实是一个`(value,storage_key)`的B+树，以便支持高效的事务性插入与更新。

#RS/WS在使用上的关系
A.插入(Insert)
所有的数据直接进入WS，此时数据并不压缩，方便事务操作。

B.删除(delete)
由于RS是只读且压缩的，物理删除的代码很大，因此C-Store引入了DRV(Deleted Record Vector):
- 逻辑删除：在RS中删除一条记录时，并不触碰RS本身，而是在WS中维护一个DRV（一个位图）里标记删除。
- 查询合并：当执行查询时，执行引擎会同时读取RS和WS中的数据，并根据DRV过滤掉RS中已经被删除的数据。

C.更新(Update)
更新操作被拆解为：Insert(新的值进WS) , Delete(增加旧值的删除标记)。

注意：C-Store假设有，查询是海量的，但只涉及少数列、数据是成批导入或偶尔更新的。
在我看来C-Store可能更适用于那种中小型产品，因为在论文中并没有提到因为高并发场景下的写、数据同步问题（这在现在的市场来看是非常正常的需求），如果出现高并发场景的写，那么它的合并会非常频繁，这会间接导致系统不可用。


#### 面向行的执行的几种物理设计方案

**垂直分区**：该方案的目的是将一张宽表拆成多个只有**两列**的窄表，每张窄表存`(position,列值)`。
缺点：
- 每列都要额外存储一个position，冗余
- 行存储每个 tuple 都有 header（记录元数据，如事务ID、空值标记等），而这些 header 对每一列都会重复存储，浪费空间
这里其实要说明下，其实上面说的`position`就是行号的意思，例如：
```
原表:
| RID | name | age | salary |
|-----|------|-----|--------|
|  1  | Tom  |  25 |  5000  |
|  2  | Bob  |  40 |  8000  |

拆分后:
age表:            salary表:
| position | age | | position | salary |
|----------|-----| |----------|--------|
|    1     |  25 | |    1     |  5000  |
|    2     |  40 | |    2     |  8000  |
```
为啥还说header浪费呢，其实就是因为当进行了拆表之后，每张表的tuple其实也还是会有header，而这些header其实只需要有一个就够了，因此对于空间来说存在极大的浪费。

**仅索引扫描法：** 索引扫描法就比较纯粹了，原先的行存储依然如此，但是会对每个列添加一个非聚集B+树索引，目的就是不要去访问实际的data page。
但是这样会有个弊端，那就是非谓词列的获取会引起索引扫描，这个扫描的效率可能会比扫描堆文件的效率低。
这里有两种场景：
- 谓词列： 查询条件（age > 10）
- 非谓词列： 投影
为什么会说非谓词列的扫描可能会被扫描堆文件更慢呢，下面的例子：
```
SELECT AVG(salary) FROM emp WHERE age > 40
```
如果age与salary都有各自的索引：
- 扫描age得到满足谓词条件的RID列表
- 拿着这些RID**逐个**去salary索引中查询对应的值（随机访问）
- 计算AVG
慢就慢在这个**随机访问**！有什么解决办法？ 答：**复合键索引！**
如果存在`Age + Salary`的复合键索引，那在谓词查询的时候我们就可以拿到`Salary`的对应值，这样就可以避免随机查询。


**实体化视图**: 将查询结果存储在磁盘上！
```
CREATE MATERIALIZED VIEW emp_mv AS
SELECT age, salary FROM emp WHERE age > 40;
-- 结果被物理存储下来，查询时直接读，不重新计算
```
论文的思路是：既然我提前知道会跑哪些查询，那就**针对每个查询预先建一张只包含它所需列的实体化视图**。(尽管该方法要求预先知晓查询工作负载，仅适用于有限场景，但是论文中说**预计其性能会优于另外两种方法**)。这也是它的缺点所在，要提前知晓workload。



#### 面向列的提升性能的优化方式

**压缩**： 直观来看列存储数据比行存储数据更具有可压缩性。压缩算法在信息熵低（数据值局部性高）的数据上表现极佳。 行存储的压缩与列存储的压缩均适用于某一列已排序或者二次排序且该列中存在相同值连续重复的情况。

**延迟物化：** 其实在我看来啊，这个名字真tm迷的很；物化是啥意思，当我们读取了每一列的数据之后我们需要将这每一列的数据按照顺序组装成元组，这个过程要做**元组构建**, 而行存储是天生就构建好的，因为行存储每一行的数据都紧凑的存在一起。因此延迟物化是不是就是延迟元组组装的意思？下面我们来看个示例：
```
SELECT name FROM emp WHERE age > 40 AND salary > 6000

三列数据长这样（列存储，每列单独存）：
age列:    [25, 40, 55, 30, 60]   ← 位置 0,1,2,3,4
salary列: [5000, 8000, 9000, 4000, 7000]
name列:   [Tom, Bob, Amy, Joe, Sue]

## 列存储怎么执行这个查询？

**第一步：对 age 列应用谓词 age > 40**
age:    [25,  40,  55,  30,  60]
满足?    否    否   是    否   是
位置列表: [2, 4]

**第二步：对 salary 列应用谓词 salary > 6000**
salary: [5000, 8000, 9000, 4000, 7000]
满足?     否    是    是    否    是
位置列表: [1, 2, 4]


**第三步：两个位置列表取交集**
age的位置列表:    [2, 4]
salary的位置列表: [1, 2, 4]
交集:             [2, 4]


如果用**位串**表示会更直观：

age    的位串: 0 0 1 0 1
salary 的位串: 0 1 1 0 1
按位与结果:    0 0 1 0 1  → 位置 [2, 4]


**第四步：用位置列表去 name 列取值**

name列: [Tom, Bob, Amy, Joe, Sue]
取位置2和4:        Amy,       Sue
```
这就是延迟物化的含义： 整个过程中，数据一直以列的形式存在，直到最后一步才把需要的值提取出去，从来没有把行拼回来过（projection之前）。
位置列表的三种形式：

| 情况         | 表达方式 | 例子         |
| ---------- | ---- | ---------- |
| 满足条件的行很少   | 简单数组 | [2,4]      |
| 瞒住条件的行很多   | 位串   | 00101      |
| 满足条件的是连续范围 | 范围集合 | [2-4,6-10] |
这么做的好处是避免无效的tuple组装: 假如有三张表进行连接，传统链式做法是先join第一张和第二张表，但是如果因为第三张表的结果是空的，这就会导致前面的join组装的tuple是无效的，浪费了资源。



**块迭代：** 在行存储中，我们需要不断的变化offset去获取tuple中的属性值，而列存储在获取列数据时的效率要更高效，块迭代像表达的就是如果一个列的数据长度是固定的，那么我们可以直接按照一个块去获取这一批的列数据，并且因为长度固定我们还可以像遍历数组一样去遍历列的值。
行存储的传统方式：Volcano模型通过Next方法一次获取一行，然后计算offset获取列值。
列存储：一次获取多个列值，通过数据遍历的形式去获取列值，不需要计算offset；数组遍历的另外一个好处就是`CPU流水线`：
```
for i in range(1000):
	result += arr[i]
```
CPU能预测下一次循环要访问`arr[i+1]`，因此它会提前把数据加载好，几乎不用等待。


**不可见连接：** 不可见连接主要是体现在多表连接时，列存储可以充分的利用延迟物化的优势，避免不必要的资源浪费。
```
SELECT c.nation, s.nation, d.year, sum(lo.revenue)
FROM lineorder lo, customer c, supplier s, dwdate d
WHERE lo.custkey = c.custkey
AND c.region = 'ASIA'
AND lo.suppkey = s.suppkey  
AND s.region = 'ASIA'
AND d.year BETWEEN 1992 AND 1997

传统做法是**流水线式连接**：

先join lineorder和customer → 过滤出亚洲客户的订单
     ↓
再join结果和supplier → 过滤出亚洲供应商
     ↓
再join结果和date → 过滤年份
     ↓
聚合


问题：每次join都要**提前把行拼起来**，失去了列存储延迟物化的优势。

---

## 不可见连接：三个阶段

### 第一阶段：在维度表上执行谓词，建哈希表

对每张维度表，先把 WHERE 条件用上，提取满足条件的**主键列表**，存成哈希表。

customer表 → 筛选 region = 'ASIA' → 满足条件的 custkey: {1, 3}
supplier表 → 筛选 region = 'ASIA' → 满足条件的 suppkey: {1}
date表     → 筛选 year in [1992,1997] → 满足条件的 datekey: {01011997, 01021997, 01031997}


三个哈希表建好，放内存里（维度表小，放得下）。

---

### 第二阶段：用哈希表在事实表外键列上打标记，得到位图

拿着哈希表，去扫事实表的外键列，看每一行是否满足条件：

事实表:
| orderkey | custkey | suppkey | orderdate  | revenue |
|    1     |    3    |    1    | 01011997   |   131   |
|    2     |    1    |    2    | 01021997   |    1    |  
|    3     |    2    |    1    | 01021997   |   456   |
...

custkey列 用哈希表{1,3}探测:  1,0,0,1,0,0,1 （位图）
suppkey列 用哈希表{1}探测:    1,0,1,0,1,1,1 （位图）
orderdate列 用哈希表探测:     1,1,1,1,1,1,1 （位图）

三个位图按位与(AND):
最终位图:                     1,0,0,1,0,0,1


得到满足**所有条件**的事实表行的位置列表 P。

---

### 第三阶段：用位置列表 P 去维度表取值

现在知道事实表里哪些行满足条件了，用这些行的外键去维度表取需要的属性：

满足条件的行的 custkey → 去 customer 表取 nation
满足条件的行的 suppkey → 去 supplier 表取 nation
满足条件的行的 orderdate → 去 date 表取 year


因为维度表的主键通常是从1开始的连续整数，所以外键值**直接就是数组下标**，取值就是一次数组查找，非常快。

---

## 为什么叫"不可见"？

因为整个过程**join操作被伪装成了谓词过滤**。

传统思维: lineorder JOIN customer ON custkey → 再过滤 region='ASIA'
不可见连接: 直接在 lineorder.custkey 列上打标记"这个值在亚洲客户集合里吗？"
```
从外面看好像只是在做列过滤，join被藏起来了，所以叫"不可见"。 为什么join被藏起来了？ 因为当join表时，实际执行的是这个表的谓词过滤操作，然后将结果做成了一个Hash表用于后续进行连接。所以这里并没有进行实际的join，因此叫做不可见连接，哈哈哈哈哈。

核心优势总结：

|      | 传统join   | 延迟物化join | 不可见连接     |
| ---- | -------- | -------- | --------- |
| 何时拼行 | join时立刻拼 | 最后拼      | 最后拼       |
| 乱序提取 | 多        | 多        | 最少        |
| 并行度  | 低        | 低        | 高（各谓词可并行） |
关键在于：**等所有谓词都用位图AND完之后，才去取维度表的值**，这时候满足条件的行已经很少了，**乱序提取的代价被压到最低。**
注意：论文中提到一个优化的点，假如瞒住条件的**主键恰好是连续的**，比如`{100,101,102,....,130}`，就不需要哈希表了，直接变成一个区间判断：
```
不用哈希表探测，直接判断：1000 <= curtKey <= 130
```


#结论
论文中在数据仓库基准测试SSBM上对比了C-Store与一款商业行存储系统的多个变体的性能。发现，通过垂直分区、仅索引计划等技术再行存储中模拟列存储的物理布局，无法实现良好的信念。论文中将这种性能低下归因为元组重构成本高昂，以及窄表和垂直分区表中每个元组的高额开销。
论文中剖析了列存储能够高效处理列存数据的原因，发现延迟物化可将性能提升三倍，压缩技术平均可带来两倍的性能提升，而在访问排序数据的查询中，压缩的性能提升幅度可达到一个数量级，此外，不可见连接的新型技术，该技术可进一步将性能提升约50%；
- **延迟物化（Late Materialization）**：贡献最大，能提升约 **3 倍** 性能 。
- **压缩（Compression）**：平均提升 **2 倍**，但在排序列上能达到一个数量级（10倍）的提升 。
- **不可见连接（Invisible Join）**：提升约 **1.5 倍** 。
- **块迭代（Block Iteration）**：提升约 **1.5 倍** 。
