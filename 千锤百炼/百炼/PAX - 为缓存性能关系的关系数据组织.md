关系型数据库系统传统上针对I/O性能进行优化，并采用N-ary存储模型（NSM，又称槽页）将记录按照顺序组织在磁盘页上。
在现代平台上，缓存利用率和性能正变得愈发重要。
在这篇论文中，它首先就证明了，页内数据布局是实现高缓存性能的关键。而NSM在现代平台上的缓存利用率较低。
PAS（属性跨分区），该模型通过将每个属性的所有值集中在每个页内，显著提升缓存性能。由于PAX仅影响页内的布局方式，因此不会带来存储开销，也不会改变IO行为。
实验结果表明，与NSM相比：
- PAX在缓存和内存带宽利用率上表现更优，可减少至少75%因数据缓存访问导致的NSM停顿时间
- 对内存驻留关系的范围选择查询和更新操作执行速度提升17% ~ 25%
- 涉及I/O的TPC-H查询执行速度提升11% ~ 48%。

论文中有提到：近期研究表明，决策支持系统，空间应用等现代数据库工作负载的性能瓶颈往往源于处理器和内存子系统的延迟，而非I/O。而`cache miss`是主要的内存瓶颈。无用数据加载的缓存会导致：
- 浪费带宽
- 污染缓存
- 可能迫使未来可能需要的信息被替换，从而引发更多延迟


这里其实我心中有个问题，那就是为什么论文中强调DMS在二级缓存中的优势（其实就是缓存命中率高），我的疑惑点在于为什么是二级缓存：
内存的层次 (这个层次来源于当时的论文的时间)：
```
CPU寄存器 ~1ns
L1 Cache ~4ns 几十KB
L2 Cache ~12ns 几百KB ← PAX重点关注这里
内存(DRAM) ~100ns
磁盘 ~10ms
```

为什么是L2不是L1？ 因为L1太小了，如果将页（8KB）放在L1中，最多就只能放4页，那必然会有频繁的换入换出，因此原因是：L1太小，效果反而会更差。

问题：难不成还能手动指定在L1还是L2进行缓存？
不能直接指定！Cache是CPU硬件自动管理的，程序员没有办法说"把这个数据放L1"。

其实这个问题不能这么看，我们应该换个角度；比如，CPU从L1,L2去找一个数据，如果发现都没有才会去内存中拿，那么L1中存储的肯定是比L2更近期使用过的数据，那论文为什么要说L2呢？ 因为L2是最后的关卡，只要L2有，那就说明了缓存命中(L1当然也是)。


这其实也是NSM的劣势所在，NSM（N-ary Storage Model）既按行存储（记录连续存放），NSM在I/O上表现良好，但是在CPU缓存上表现差：即使需要一个小字段，也会把整个一行都加入到缓存中，照成**缓存污染**和**带宽浪费**。

论文中提到了在实际的测试中，使用PAX存储的TPC-H表比NSM存储的表节省约8%的空间，在这8%中：
- 一半是因为消除了页面尾部的槽表
- 另外一半是NSM中每条记录的额外头

#数据插入，一个有意思的地方
其实推测在page中肯定是有多个v-miniPage和F-miniPage，对于数据的插入，论文中提到一个有意思的点，那就是假如插入可变长度数据时空间不够，那么PAX会通过计算平均可变长度来移动v-minipage来空出可用空间来；如果还不行，甚至还会从相邻miniPage借用空间，如果相邻miniPage还不够，那记录则会被移动到另外一个页面。
**这种设计的目的是保证页面填满**，避免频繁创建新页。


在第五章中严格证明了为什么PAX比NSM的缓存性能更好，排除IO，变长字段，复杂查询等因素干扰：
- 关系完全内存驻留（无I/O影响）
- 只使用固定长度数值属性
- 简单范围查询：`select avg(ap) from R where aq > lo and aq > hi`
- 记录大小、缓存行大小、属性对齐
- 用硬件性能计数器测量：L1/L2缓存miss、CPU stall时间、指令退休率

#关键发现：
- NSM：每个记录都会触发`cache miss`
- PAX：每四个记录才触发一次`cache miss`
- PAX减少75%的L2数据miss，CPU stall时间减少75%
- 投影属性越多，PAX优势会缩小（但仍优于NSM）
PAX通过页内列式存储极大提升空间局部性，从而减少缓存miss和CPU停顿。
